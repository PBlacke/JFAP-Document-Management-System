📅 Development Timeline Journal
Project: JFAP Simple Document Management System (DMS) Prototype


🗓️ February 27, 2026 – Project Kickoff & Core Implementation
Started building the prototype from scratch using Flask, SQLite, and Tesseract OCR.
Implemented basic functionality:
    - File upload (images and PDFs) with secure filename handling.
    - OCR text extraction using Tesseract (via pytesseract).
    - Storage of document metadata and extracted text in an SQLite database.
    - Simple search using LIKE queries on the extracted text.
    - Clean, responsive frontend with vanilla HTML/CSS/JavaScript.
    - Set up project structure with app.py, templates/, static/, and uploads/ folders.
    - Initialised Git repository and pushed the first working version to GitHub.
✅ Achieved: A functional DMS that can upload, OCR, store, and search documents.


🗓️ March 2, 2026 – Debugging, Documentation, and Preview Attempt
Focused on refining the existing code:
    - Fixed several typos and syntax errors (e.g., table creation, missing commas, variable name mismatches).
    - Resolved issues with Tesseract path configuration and PDF processing.
    - Corrected the favicon not showing by moving it to the static/ folder and using url_for.
    - Added a comprehensive README to the GitHub repository, detailing installation, usage, and tech stack.
    - Began implementing document preview functionality:
    - Created /preview/<int:doc_id> route to serve images and PDFs in an embedded viewer.
    - Added preview_image.html and preview_pdf.html templates.
Encountered bugs: missing doc_id in template context and SQL query errors (typo in table name, missing commas in tuples).
🐛 Challenges: Preview feature was incomplete due to template variable errors and database query issues.


🗓️ March 3, 2026 – Feature Completion, UI Polish, and Debugging
    Enhanced upload interface:
        - Added drag‑and‑drop support to the upload area (files are placed into the file input but not auto‑uploaded – user must click the button, maintaining consistent behaviour).
        - Added Document Type dropdown (PR, PO, Other) and Project text field to the upload form; these are stored in the database.
    Database schema updated:
        - Added doc_type and project columns to the documents table (with error handling for existing columns).
    Redesigned document editing:
        - Replaced the separate edit.html page with a modal popup on the documents page.
        Modal includes:
            - A preview of the document (image thumbnail or PDF icon).
            - Editable filename (without extension), document type dropdown, and project field.
            - A Delete button inside the modal (with confirmation) for convenience.
        - Updated /edit/<int:doc_id> route to handle only POST requests, updating filename, type, and project, and renaming the actual file on disk.
        - Updated /delete/<int:doc_id> route remains POST-only.
    Documents list enhanced:
        - Displays type and project under each filename.
        - Edit button (pencil icon) opens the modal; delete button (trash icon) remains inline for quick deletion (with confirmation).
    Fixed remaining bugs:
        - Corrected JavaScript errors in uploadFile() (moved FormData creation before appending fields).
        - Fixed edit modal population by retrieving data from data-* attributes, ensuring type and project are correctly set.
        - Set dynamic action for both edit and delete forms inside the modal.
        - Resolved SQL parameter tuple issues (commas) in all database calls.
        - Ensured url_for imported everywhere needed.
        - Timestamp formatting now uses standard %Y-%m-%d %H:%M:%S (Windows‑compatible).
    Tested end‑to‑end:
        - Upload with type/project works (both manual and drag‑and‑drop).
        - Search returns results; preview opens correct viewer.
        - Edit modal opens with correct data, saves changes (file renamed, database updated).
        - Delete removes both record and file.
        - All features working smoothly.
✅ Outcome: A fully functional DMS prototype with upload (with drag‑and‑drop), OCR, search, preview, inline editing modal, and delete, complete with document tagging (type/project) and a polished UI. Well‑documented on GitHub.


🗓️ March 4, 2026 – Full‑Text Search, User Authentication, and UI Polish
    Implemented SQLite FTS5 full‑text search:
        - Created documents_fts virtual table linked to the main documents table.
        - Added triggers to keep the FTS index in sync on insert, update, and delete.
        - Replaced the old LIKE‑based search with a lightning‑fast MATCH query.
        - Enhanced search with BM25 relevance ranking – results are now ordered by how well they match the query, then by date.
        - Verified that searches return whole‑word matches, support phrases, and ignore stop words.
    Added user authentication system:
        - Installed and configured Flask‑Login.
        - Created users table with fields for id, username (formatted as first.last), email (optional), and hashed password.
        - Implemented password validation: minimum 8 characters, at least one number, and one special character.
        - Built registration and login routes with proper error handling.
        - Created styled login and registration pages matching the existing UI.
        - Protected all document‑related routes with @login_required.
        - Modified every database query to filter documents by current_user.id, ensuring each user sees only their own documents.
        - Updated navigation links to show logout, username, or login/register depending on authentication state.
    Root route behavior changed:
        - The home page now redirects unauthenticated users to the login page; authenticated users go directly to the upload/search interface.
    Bug fixes and refinements:
        - Fixed SQL parameter binding errors in multiple routes (e.g., list_documents, preview_document) by ensuring correct number of placeholders and passing current_user.id.
        - Corrected unpacking errors in preview_document – selected only the needed columns.
        - Added ownership verification to all document‑specific endpoints (/edit, /delete, /preview, /view) to prevent cross‑user access.
        - Tidied up the UI for login and registration with consistent styling and helpful hints.
✅ Outcome: The JFAP DMS Prototype now boasts multi‑user support, fast and relevant full‑text search, and a cohesive, secure experience. All core features (upload, OCR, search, preview, edit, delete) are locked to individual users. The system is ready for further enhancements like email verification or deployment.


🗓️ March 5, 2026 – UI/UX Enhancements, and Tweaks
    Implemented a consistent navigation bar using template inheritance (base.html):
        - Navbar includes logo, Home, Documents links, and a profile dropdown with user's full name and email.
        - Added active page highlighting.
        - Integrated logout link with a custom icon.
    Added a simple footer across all pages: "Developed by me for JFAP Construction".
    Redesigned document preview page:
        - Split layout into two columns: left side shows the document (image or PDF), right side displays a details panel.
        - Panel includes document type, project, uploader's username, upload date, and a prominent "Download Original" button styled like site buttons.
        - Fixed a bug where the preview route selected too few columns (caused unpack error) – now correctly fetches all metadata.
    Database housekeeping:
        - Noted that old files (uploaded before user authentication) had NULL user_id and were not showing in the documents list.
        - Advised on updating them with UPDATE documents SET user_id = ? WHERE user_id IS NULL to associate with the current user.
✅ Outcome: The JFAP DMS Prototype now boasts a polished, professional interface with consistent navigation, and a user‑friendly preview panel. All core features remain stable and secure.
