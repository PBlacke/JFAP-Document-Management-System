🗓️ February 27, 2026 – Project Kickoff & Core Implementation
Started building the prototype from scratch using Flask, SQLite, and Tesseract OCR.
Implemented basic functionality:
    - File upload (images and PDFs) with secure filename handling.
    - OCR text extraction using Tesseract (via pytesseract).
    - Storage of document metadata and extracted text in an SQLite database.
    - Simple search using LIKE queries on the extracted text.
    - Clean, responsive frontend with vanilla HTML/CSS/JavaScript.
    - Set up project structure with app.py, templates/, static/, and uploads/ folders.
    - Initialised Git repository and pushed the first working version to GitHub.
✅ Achieved: A functional DMS that can upload, OCR, store, and search documents.

🗓️ March 4, 2026 – Full‑Text Search, User Authentication, and UI Polish
    Implemented SQLite FTS5 full‑text search:
        - Created documents_fts virtual table linked to the main documents table.
        - Added triggers to keep the FTS index in sync on insert, update, and delete.
        - Replaced the old LIKE‑based search with a lightning‑fast MATCH query.
        - Enhanced search with BM25 relevance ranking – results are now ordered by how well they match the query, then by date.
        - Verified that searches return whole‑word matches, support phrases, and ignore stop words.
    Added user authentication system:
        - Installed and configured Flask‑Login.
        - Created users table with fields for id, username (formatted as first.last), email (optional), and hashed password.
        - Implemented password validation: minimum 8 characters, at least one number, and one special character.
        - Built registration and login routes with proper error handling.
        - Created styled login and registration pages matching the existing UI.
        - Protected all document‑related routes with @login_required.
        - Modified every database query to filter documents by current_user.id, ensuring each user sees only their own documents.
        - Updated navigation links to show logout, username, or login/register depending on authentication state.
    Root route behavior changed:
        - The home page now redirects unauthenticated users to the login page; authenticated users go directly to the upload/search interface.
    Bug fixes and refinements:
        - Fixed SQL parameter binding errors in multiple routes (e.g., list_documents, preview_document) by ensuring correct number of placeholders and passing current_user.id.
        - Corrected unpacking errors in preview_document – selected only the needed columns.
        - Added ownership verification to all document‑specific endpoints (/edit, /delete, /preview, /view) to prevent cross‑user access.
        - Tidied up the UI for login and registration with consistent styling and helpful hints.
✅ Outcome: The JFAP DMS Prototype now boasts multi‑user support, fast and relevant full‑text search, and a cohesive, secure experience. All core features (upload, OCR, search, preview, edit, delete) are locked to individual users. The system is ready for further enhancements like email verification or deployment.

🗓️ March 9, 2026 – Performance Optimization, Bottleneck Hunting, and Stabilisation
    Focused on diagnosing and improving system performance after noticing slow uploads and deletions.
    Added detailed timing logs to /upload, /delete, and /documents routes to pinpoint slow operations.
    Enabled Write‑Ahead Logging (WAL) on all database connections to reduce locking and improve concurrency.
    Created database indexes on frequently queried columns: documents(user_id), documents(upload_date), document_versions(doc_id), and activity_log(user_id) – drastically speeding up queries.
    Optimised the /upload route:
        Separated main document insert from log_activity to avoid contention.
        Reduced total upload time from ~31 seconds to ~2 seconds (with OCR still the main contributor).
    Investigated slow deletions:
        Discovered that the delay was actually in the redirect to /documents, not the delete itself.
        Added timing to list_documents and confirmed query performance improved with new indexes.
    Encountered new bugs (unexpected behaviour in delete and list views) – decided to pause and return on Wednesday with fresh eyes.
✅ Outcome: The system is now significantly faster and more robust. Uploads average ~2 seconds, and document listing is near‑instant even with many records. Remaining issues noted for later resolution.
